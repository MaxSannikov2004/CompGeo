# Лабораторная работа 1 — автопроверка (GitHub Actions)

Этот репозиторий настроен так, чтобы при каждом `push` запускались автотесты, и вы сразу видели, проходит ли решение.

## Как сдаёт студент

1) Откройте ноутбук **`Лабораторная_работа_1.ipynb`** и реализуйте функции:

- `orient(A, B, C)`
- `dist_point_line(P, a, b, c)`
- `on_segment(A, B, P)`
- `segments_intersect(A, B, C, D)`
- `polygon_area(poly)`
- `point_in_polygon(Q, poly)`

> Можно писать и в `lab1.py` (если в ноутбуке остались заглушки, тесты попробуют импортировать `lab1.py`).

2) Локальная проверка (рекомендуется):
```bash
python -m pip install -r requirements.txt
pytest -q
```

3) Загрузите изменения на GitHub (`git push`).
- Вкладка **Actions** покажет результат автопроверки.
- Если тесты упали, в логах будет видно *какой именно тест* и *на каких данных*.

## Как считается “балл за код”

В CI считается **6/6 за код**: по **1 баллу за каждую функцию**.  
Job будет зелёным только при **6/6**.

Функция → 1 балл:
- orient
- dist_point_line
- on_segment
- segments_intersect
- polygon_area
- point_in_polygon

Общий курсный рубрикатор: **8 баллов**, где **код = 6**, остальное (2) — отчёт/визуализация/обоснование (вне автотестов).

## Для преподавателя: как раздать и как проверять

### Раздача
Самый удобный вариант — **GitHub Classroom**:
- создайте assignment из этого репо-шаблона;
- студенты принимают задание → получают свои репозитории;
- тесты бегут автоматически на каждый `push`.

Альтернатива: один общий репозиторий → студенты делают **fork** и открывают **Pull Request**.

### Как преподавателю проверить “чужой” код
Да:
- вы можете просто открыть репозиторий студента → вкладка **Actions** → последний запуск → логи;
- или клонировать репозиторий студента себе и запустить `pytest -q` локально.

> Примечание: тесты в репозитории видны студентам (это “public tests”).  
> Если хотите “скрытые” проверки — держите отдельный приватный набор тестов и прогоняйте его у себя.
